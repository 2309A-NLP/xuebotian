# Qwen3-VL-2B 优化文档

## 1. 当前状态

基于现有训练前后评估结果，模型已经取得明显收益：

- 准确率：`16.67% -> 91.67%`
- 正确题数：`2/12 -> 11/12`
- 工业综合分：`20.94 -> 48.46`
- 图示推理得分：`0.3215 -> 0.6167`

当前优化工作的目标不是“从零提升效果”，而是解决剩余缺陷、增强泛化、降低部署风险。

## 2. 已识别问题

### 2.1 单样本推理异常

唯一残留错误样本是：

- `CN100347506C_002`

训练前后都出现同样的异常：

- `ERROR: index 1 is out of bounds for dimension 0 with size 1`

这说明问题更偏向以下几类：

- 图像输入数量与模型输入张量维度不一致
- processor 输出与解码逻辑存在边界条件缺陷
- 某条样本的消息结构、图片字段或文本字段异常

### 2.2 Linux 复现风险

训练数据和部分评估数据使用了 Windows 绝对路径，例如：

- `G:\...\images\CN100342976C\page1.png`

这会导致：

- Linux 无法直接训练
- Linux 无法直接评估
- 容器化部署时数据目录不可移植

### 2.3 模板配置不完全统一

当前配置中出现了模板不一致：

- 训练配置：`qwen3_vl_nothink`
- 合并配置：`qwen3_vl`

虽然不一定立即导致错误，但会带来以下风险：

- 训练与推理的消息格式不一致
- 评估表现与真实部署表现不一致
- 后续问题定位难度增加

### 2.4 数据规模偏小

当前评估样本只有 `12` 题，训练配置还带有：

- `max_samples: 1000`

这意味着当前结果更像是“已在小规模数据上证明路线有效”，但还不能证明模型对更广专利类型的泛化能力。

### 2.5 训练轮数偏高，存在过拟合风险

当前训练参数为：

- `learning_rate: 2e-4`
- `num_train_epochs: 20`

对于 2B 模型 + 1000 样本量级的数据集，这个组合偏激进，可能出现：

- 记忆化回答
- 训练集模式学习过强
- 对未见专利的泛化变差

## 3. 优化建议

## 3.1 P0：优先修复项

### 3.1.1 排查 `CN100347506C_002` 异常

建议重点检查：

- 该样本 `images` 字段是否为空、重复或路径失效
- `processor(text=[...], images=images, ...)` 在单图场景下的返回张量形状
- `model.generate` 前后的 batch 维度
- `processor.batch_decode(outputs, ...)` 是否处理了非标准输出

建议直接对该样本单独做最小复现脚本，打印：

- 样本原始内容
- 图片路径
- 图片加载数量
- `inputs` 的各项 shape
- `outputs.shape`

### 3.1.2 统一模板

建议将训练、合并、推理统一为：

- `qwen3_vl_nothink`

原因：

- 当前训练已经基于该模板取得好结果
- 推理模板应和训练模板保持一致
- 可以减少输入格式偏差

### 3.1.3 解决 Linux 路径问题

建议不要直接在 Linux 上复用 `*_windows.json` 数据文件，而是：

- 生成 Linux 版训练数据文件
- 生成 Linux 版评估数据文件
- 使用环境变量控制图片根目录映射

这项工作已经体现在新增的 `linux_deploy.sh` 中。

## 3.2 P1：训练侧优化

### 3.2.1 调整学习率与 epoch

建议做两组对比实验：

- 方案 A
  - `learning_rate: 1e-4`
  - `num_train_epochs: 8~12`
- 方案 B
  - `learning_rate: 5e-5`
  - `num_train_epochs: 10~15`

目标是看是否能在保持高准确率的同时，降低对特定表述的过拟合。

### 3.2.2 扩大验证强度

建议：

- 保留 `val_size`
- 记录每个 `eval_steps` 的最优 checkpoint
- 不只看最终 epoch

如果后续样本量扩大，建议引入：

- early stopping
- best checkpoint export

### 3.2.3 增强困难样本

建议围绕当前题型补充数据：

- 图示空间关系反向问法
- 同一图纸多部件对比问法
- 公式符号含义映射题
- 术语近义干扰题
- 结构化 JSON 输出题

重点补强两类弱点：

- 工业术语稳定命中
- 复杂图示推理链路

## 3.3 P1：评估侧优化

### 3.3.1 扩大评估集

建议把评估集从 `12` 题提升到至少：

- 小规模回归集：`50` 题
- 稳定验证集：`100+` 题

并按类型分桶：

- 文本抽取
- 选择判断
- 图示定位
- 流向推理
- 公式解释

### 3.3.2 增加异常统计

当前评估更偏效果指标，建议新增：

- 推理异常率
- 图片加载失败率
- 空回答率
- 非结构化输出率

这样能把“回答错”和“程序异常”区分开。

### 3.3.3 固化回归样本集

建议至少把以下样本纳入固定回归：

- `CN100347506C_002`
- 所有曾经从错变对的关键样本
- 所有图示推理题

这样后续调参不会悄悄破坏现有效果。

## 3.4 P2：代码与工程优化

### 3.4.1 评估脚本增加健壮性保护

建议在 `evaluate_model.py` 中增加：

- 图像不存在时的显式告警和跳过策略
- 对单样本异常的更细粒度日志输出
- 对 `messages` / `conversations` 结构的校验
- 运行前参数合法性检查

### 3.4.2 输出文件校验

建议在生成 `evaluation_report.json` 后增加一次自校验：

- 重新读取并解析 JSON
- 如果失败则另存 raw 结果

避免后处理阶段出现“结果文件存在但无法解析”的问题。

### 3.4.3 拆分运行配置

建议分别维护：

- `train_*`
- `merge_*`
- `infer_*`
- `eval_*`

避免一个配置文件兼顾多个阶段，降低维护成本。

## 4. 推荐执行顺序

建议按以下顺序推进：

1. 修复 `CN100347506C_002` 异常
2. 统一模板为 `qwen3_vl_nothink`
3. 生成 Linux 可复现数据和配置
4. 补充 50 题回归集
5. 做两组学习率 / epoch 对比实验
6. 固化最优训练与部署流程

## 5. 预期收益

完成上述优化后，预期可以获得以下收益：

- 将当前单样本异常清零
- 降低 Linux 部署和复现阻力
- 提升图示推理与术语命中的稳定性
- 减少过拟合风险
- 为后续更大规模评估和上线提供标准化流程

## 6. 结论

当前模型已经证明“路线正确”，下一阶段重点应从“继续堆指标”转向“修异常、保稳定、补泛化、做标准化”。从工程价值看，最优先的是解决单样本推理异常和 Linux 数据路径问题，这两项解决后，整套流程才真正具备可迁移和可部署性。
