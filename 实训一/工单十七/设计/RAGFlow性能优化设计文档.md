# RAGFlow性能优化设计文档

## 1. 文档说明

本文档基于 `压测结果/优化操作` 目录下已有优化材料整理，目标是形成一份统一的设计文档，用于说明 RAGFlow 在以下两条关键链路上的性能优化设计：

- PDF 上传链路性能优化
- 检索 + 生成请求链路性能优化

本文档聚焦设计思路、改造范围、关键实现、配置方案、风险控制与验证方法，不展开代码逐行说明。

## 2. 背景与问题

当前系统部署在 Docker 容器中，压测与问题定位显示，性能瓶颈主要集中在两类场景：

1. 文档上传场景中，大文件 PDF、多文件批量上传、URL 导入会产生明显时延。
2. 检索 + 生成场景中，高并发请求会放大 I/O 等待、下游 LLM 波动、租户间抢占与突发流量问题。

现状问题如果不处理，会带来以下影响：

- 上传耗时过长，超时概率升高
- 多文件上传吞吐偏低
- MinIO 存储写入成为瓶颈
- URL 下载分支阻塞事件循环
- Chat 请求在高并发下缺少限流与熔断保护
- 某一租户的突发流量可能影响其他租户

## 3. 设计目标

本次优化的目标如下：

- 降低 PDF 上传链路端到端耗时
- 提升多文件上传和多 URL 导入场景的并发吞吐
- 减少同步 I/O 对主链路的阻塞
- 为检索 + 生成链路增加限流、熔断与并发控制能力
- 实现多租户隔离，避免热点租户拖垮整体服务
- 通过环境变量控制关键开关，支持灰度发布和快速回退

## 4. 设计范围

本次设计包含以下范围：

- 上传 API 链路
- 文件服务层 PDF 处理逻辑
- MinIO 写入路径
- URL 下载路径
- Chat API 请求入口
- 异步限流与熔断中间件
- Docker 环境变量配置

本次设计不包含以下内容：

- 检索召回算法本身的质量优化
- LLM 模型替换或提示词策略优化
- MinIO 集群架构改造
- 前端交互改造

## 5. 总体设计

### 5.1 PDF 上传链路

当前上传链路如下：

```text
客户端请求
  -> Quart 中间件
  -> /documents/upload API
  -> FileService.upload_info()
  -> 分支一: URL 下载 / 分支二: 文件读取
  -> PDF 预检验
  -> put_blob() 写入 MinIO
  -> 返回元数据
```

优化后的核心思路：

- 放宽上传相关超时与请求体限制，避免大文件被非业务性超时打断
- 多文件上传由串行改为并行
- MinIO 写入改为异步执行，并保留并发上限控制
- PDF 预检验改为可配置开关，在需要时跳过上传阶段校验
- 多 URL 下载改为并发执行，避免逐个阻塞

### 5.2 检索 + 生成链路

当前痛点不在单一函数，而在高并发时缺少统一入口控制。优化后的总体思路如下：

```text
客户端请求
  -> Chat API
  -> 限流装饰器
     -> 熔断器检查
     -> 租户级限流检查
     -> 必要时等待可用配额
  -> 原始 async_chat 逻辑
     -> 检索
     -> Prompt 构建
     -> LLM 生成
```

设计重点：

- 在 API 入口前置保护，优先阻断异常流量
- 用租户级限流防止局部热点影响全局
- 用熔断器防止下游异常导致请求雪崩
- 通过异步方式降低线程阻塞和排队放大

## 6. 详细设计

### 6.1 上传链路优化设计

#### 6.1.1 配置项优化

针对大文件和慢链路场景，调整以下参数：

| 参数 | 建议值 | 作用 |
|------|------|------|
| `MAX_CONTENT_LENGTH` | `2147483648` | 将上传体限制提升至 2GB |
| `QUART_RESPONSE_TIMEOUT` | `1200` | 延长响应超时时间 |
| `QUART_BODY_TIMEOUT` | `1200` | 延长请求体读取超时时间 |
| `MAX_CONCURRENT_MINIO` | `20` | 提高 MinIO 写入并发能力 |

设计原则：

- 先通过配置扩容解决显式超时问题
- 配置调整不改变业务逻辑，适合作为第一阶段优化
- 所有值都保留为环境变量，方便按机器规格调整

#### 6.1.2 多文件并行上传

现状是多个文件通过列表推导逐个 `await`，总耗时接近各文件处理时间之和。

优化方案：

- 将串行 `await` 改为 `asyncio.gather`
- 让多个文件在同一批次中并发执行上传处理
- 保留异常收集能力，避免单文件失败中断整批任务

预期收益：

- 批量上传场景吞吐明显提升
- 单请求内的总耗时下降

实施要点：

- 对返回结果统一做异常归集
- 避免无限并发，必要时增加批次或信号量限制

#### 6.1.3 MinIO 写入异步化

现状 `put_blob()` 为同步调用，上传请求会直接等待对象存储写入完成。

优化方案：

- 提供异步版本 `put_blob_async()`
- 通过 `run_in_executor` 将阻塞式写入放入线程池执行
- 配合 `MAX_CONCURRENT_MINIO` 控制并发上限

设计价值：

- 减少主协程等待阻塞
- 避免高并发时所有请求都卡在存储写入点

实施注意：

- 存储层仍为同步 SDK 时，异步化本质是线程池隔离
- 并发值过高会造成连接池、磁盘或网络争用

#### 6.1.4 PDF 预检验开关化

现状上传阶段对 PDF 执行 `read_potential_broken_pdf()`，会增加一次额外处理。

优化方案：

- 新增 `SKIP_PDF_PRECHECK` 环境变量
- 上传阶段按配置决定是否跳过预检验
- 如果跳过，则把可读性校验责任后移到解析阶段

适用场景：

- 对上传速度要求高于即时校验的场景
- 内部受控数据源或可容忍解析阶段失败的场景

风险：

- 损坏 PDF 可能在上传时通过，但会在后续解析时失败

#### 6.1.5 URL 下载并发化

现状 URL 分支使用 `asyncio.run(adownload())`，会阻塞当前事件循环，多个 URL 无法自然并发。

优化方案：

- 将单 URL 下载改为异步任务列表
- 通过 `asyncio.gather` 并发拉取多个 URL
- 对单个 URL 失败做容错，不影响其余任务

收益：

- URL 批量导入的总耗时显著降低
- 更符合当前异步框架的执行模型

### 6.2 检索 + 生成链路优化设计

#### 6.2.1 异步请求队列与并发控制

设计中引入异步基础模块，承担以下职责：

- 信号量控制最大并发数
- 队列管理等待请求
- 为不同优先级请求预留扩展能力

核心价值：

- 控制系统同时处理的请求数
- 避免高峰流量直接压满服务
- 为后续按优先级调度留出实现空间

#### 6.2.2 租户级限流器

限流器采用租户维度进行统计，核心参数包括：

| 参数 | 默认值 | 说明 |
|------|------|------|
| `CHAT_RPM_LIMIT` | `60` | 每租户每分钟请求数 |
| `CHAT_RPS_LIMIT` | `10` | 每租户每秒请求数 |

设计机制：

- 使用令牌桶或等价时间窗口思路做请求准入
- 当请求超限时，允许等待短时间配额恢复
- 超过等待阈值后返回 `429 Too Many Requests`

设计收益：

- 抑制瞬时流量尖峰
- 实现租户间隔离
- 为容量规划提供可控上限

#### 6.2.3 熔断器保护

熔断器用于保护下游 LLM 或生成链路，避免故障放大。

关键参数：

| 参数 | 默认值 | 说明 |
|------|------|------|
| `CIRCUIT_BREAKER_FAILURE_THRESHOLD` | `5` | 连续失败阈值 |
| `CIRCUIT_BREAKER_RECOVERY_TIMEOUT` | `30` | 熔断恢复探测间隔，单位秒 |

状态流转：

```text
CLOSED -> OPEN -> HALF_OPEN -> CLOSED
```

执行策略：

- `CLOSED`：正常放行，累计失败数
- `OPEN`：直接拒绝请求，快速失败返回 `503`
- `HALF_OPEN`：放少量探测流量，成功后恢复，失败则重新熔断

设计收益：

- 避免下游异常导致请求大量堆积
- 通过快速失败缩短故障恢复时间
- 降低线程、连接和队列资源被无效请求耗尽的概率

#### 6.2.4 API 装饰器集成

在 Chat API 入口增加 `@rate_limit_chat` 装饰器，统一接入以下逻辑：

1. 检查限流是否开启
2. 检查熔断器状态
3. 检查租户级速率额度
4. 超限时等待短时间配额恢复
5. 调用原始业务逻辑
6. 根据执行结果记录成功或失败

这样可以将保护逻辑集中在入口层，避免分散在业务代码中。

## 7. 配置设计

建议在 `docker/.env` 中统一维护以下配置：

```bash
# 上传链路
MAX_CONCURRENT_MINIO=20
QUART_RESPONSE_TIMEOUT=1200
QUART_BODY_TIMEOUT=1200
MAX_CONTENT_LENGTH=2147483648
SKIP_PDF_PRECHECK=false

# 请求保护链路
RATE_LIMIT_ENABLED=true
CHAT_RPM_LIMIT=60
CHAT_RPS_LIMIT=10
CIRCUIT_BREAKER_FAILURE_THRESHOLD=5
CIRCUIT_BREAKER_RECOVERY_TIMEOUT=30
```

同时在 `docker-compose.yml` 中透传关键环境变量，确保容器运行时可读取。

## 8. 实施方案

建议按两阶段推进：

### 阶段一：低风险配置优化

- 调整上传大小限制与超时配置
- 调整 MinIO 并发参数
- 开启请求保护相关环境变量

这一阶段优先解决显式超时和容量过小问题，改动风险低，回退简单。

### 阶段二：代码级并发与保护改造

- 多文件上传改为并发执行
- MinIO 写入改为异步线程池执行
- URL 下载改为并发任务
- 接入 Chat API 限流装饰器
- 引入熔断器与异步请求控制模块

这一阶段对吞吐和稳定性收益更大，但需要经过联调与压测验证。

## 9. 预期效果

基于现有优化记录，预期收益如下：

| 优化项 | 预期提升 |
|------|------|
| 并行文件处理 | 50% 以上，主要体现在多文件上传 |
| MinIO 并发写入 | 20% 到 30% |
| 跳过 PDF 预检验 | 10% 到 20% |
| URL 下载并发 | 60% 以上，主要体现在多 URL 场景 |
| Chat API 限流与熔断 | 稳定性显著提升，降低雪崩风险 |

说明：

- 上传链路的收益更容易通过压测直接量化
- 检索 + 生成链路的收益更偏稳定性、失败率和尾延迟改善

## 10. 风险与控制措施

### 10.1 PDF 预检验关闭风险

风险：

- 损坏文件在上传阶段未被拦截

控制措施：

- 默认保持 `SKIP_PDF_PRECHECK=false`
- 仅在明确追求吞吐且解析阶段有兜底校验时开启

### 10.2 并发过高风险

风险：

- MinIO 连接池耗尽
- 容器 CPU、内存和网络带宽抖动

控制措施：

- 对 MinIO 并发数设置上限
- 根据压测结果逐级调参，不一次性放大

### 10.3 误限流风险

风险：

- 正常高峰业务被限制

控制措施：

- 优先按租户维度配置
- 对 `RPM`、`RPS` 分开观测
- 保留关闭开关 `RATE_LIMIT_ENABLED`

### 10.4 熔断误触发风险

风险：

- 短期异常导致服务快速进入熔断

控制措施：

- 调整失败阈值与恢复时间
- 观察 LLM 调用失败类型，区分业务异常和系统异常

## 11. 验证方案

### 11.1 上传链路验证

- 修改配置后重启容器
- 使用大 PDF 文件验证单文件上传时延
- 使用多文件批量上传验证总耗时与吞吐
- 使用多 URL 导入验证并发下载效果
- 通过 `docker stats` 和 MinIO 日志观察资源利用率与写入行为

### 11.2 检索 + 生成链路验证

- 验证限流模块和熔断模块能正常导入
- 对 Chat API 做突发请求压测
- 观察正常情况下是否有误限流
- 模拟下游失败，确认是否触发 `503`
- 验证熔断恢复后请求是否能重新放行

### 11.3 重点观测指标

- 上传成功率
- 上传平均时延与 P95/P99
- 多文件上传总耗时
- MinIO 写入耗时
- Chat 接口成功率
- Chat 接口 P95/P99
- `429` 比例
- `503` 比例
- 容器 CPU、内存、网络、线程池占用

## 12. 回退方案

如果上线后出现异常，可按以下方式快速回退：

- 恢复原始超时和上传大小配置
- 将 `RATE_LIMIT_ENABLED=false`
- 将 `SKIP_PDF_PRECHECK=false`
- 将并发参数恢复到原始值
- 暂时移除 Chat API 入口装饰器
- 必要时回退到原始上传串行逻辑

回退原则：

- 优先通过环境变量关闭功能
- 只有在配置回退无效时再回退代码

## 13. 结论

本次优化设计分为上传链路提速和请求链路稳态保护两个方向。前者解决吞吐和时延问题，后者解决高并发下的可用性和隔离性问题。整体方案采用“配置先行、代码跟进、可灰度、可回退”的策略，适合在现有 Docker 部署架构下逐步落地。
