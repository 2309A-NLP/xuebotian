# Embedding 微调评估结果

## 1. 数据集与训练设置

本次实验使用 `generated_qa_pairs.csv` 作为训练与评估数据源。数据读取后进行了空值过滤，并按照训练集、验证集、测试集进行划分。

| 数据集 | 样本数 |
| --- | ---: |
| 训练集 | 124 |
| 验证集 | 15 |
| 测试集 | 15 |

训练配置如下：

| 配置项 | 取值 |
| --- | --- |
| 基础模型 | `/root/autodl-tmp/model/bge-base-zh-v1.5` |
| 训练轮数 | 1 epoch |
| 训练 batch size | 4 |
| 梯度累积步数 | 16 |
| 学习率 | 2e-5 |
| 学习率调度 | cosine |
| 损失函数 | MatryoshkaLoss + MultipleNegativesRankingLoss |
| Matryoshka 维度 | 768, 512, 256, 128, 64 |

## 2. 验证集训练前后对比

验证集使用 `nDCG@10` 作为主要检索效果指标。数值越高，表示相关文档在检索结果前 10 名中的排序越靠前。

| Embedding 维度 | 训练前 nDCG@10 | 训练后 nDCG@10 | 绝对提升 | 相对提升 |
| ---: | ---: | ---: | ---: | ---: |
| 768 | 0.8216 | 0.9099 | +0.0883 | +10.74% |
| 512 | 0.7754 | 0.8929 | +0.1175 | +15.15% |
| 256 | 0.7964 | 0.9099 | +0.1135 | +14.25% |
| 128 | 0.7374 | 0.9099 | +0.1725 | +23.39% |
| 64 | 0.6070 | 0.8841 | +0.2771 | +45.64% |

从验证集结果看，微调后所有 Matryoshka 维度的检索效果均有提升。其中低维度提升最明显，64 维从 0.6070 提升到 0.8841，说明微调显著增强了压缩向量维度下的语义检索能力。

128 维是本次训练设置中的最佳模型选择指标，训练前为 0.7374，训练后提升到 0.9099，绝对提升 0.1725，说明模型在较低向量维度下仍能保持较好的检索排序质量。

## 3. 测试集最终结果

最终模型在测试集上的 `nDCG@10` 结果如下：

| Embedding 维度 | 测试集 nDCG@10 |
| ---: | ---: |
| 768 | 0.9333 |
| 512 | 0.9534 |
| 256 | 0.9298 |
| 128 | 0.8274 |
| 64 | 0.7786 |

测试集上 512 维取得最高结果，`nDCG@10` 为 0.9534；768 维和 256 维也保持在 0.92 以上，说明模型在较高维度下具有较稳定的检索表现。

128 维和 64 维相比高维结果有所下降，但仍达到 0.8274 和 0.7786。对于需要降低向量存储和检索成本的场景，128 维可以作为效果与资源占用之间的折中选择。

## 4. 结论

本次微调有效提升了基础 embedding 模型在当前 RAG 问答数据上的检索能力。验证集上所有维度的 `nDCG@10` 均明显提升，尤其是 64 维和 128 维，说明 Matryoshka 训练对低维向量表示有较大帮助。

最终测试集结果显示，512 维表现最好，`nDCG@10` 达到 0.9534；如果追求最高检索效果，可优先使用 512 维或 768 维向量。如果更关注存储、索引和检索速度，128 维是较合理的压缩方案。

需要注意的是，本次验证集和测试集样本数均为 15，规模较小，结果可能存在一定波动。后续可以增加数据量，并继续观察不同维度下的稳定性和泛化表现。
