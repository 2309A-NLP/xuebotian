# RAG QA System 项目已做优化点

本文档基于当前项目代码结构整理，覆盖后端架构、文档解析、文本处理、向量检索、RAG 问答、前端交互、日志异常和部署可维护性等方面，便于项目汇报、答辩和后续迭代。

## 1. 总体架构优化

### 1.1 前后端分离但部署简单

项目采用 FastAPI 作为后端服务，前端使用原生 HTML/CSS/JS。后端统一提供 `/api` 接口，同时在 `app/main.py` 中挂载 `frontend/assets` 并返回首页，因此运行 `python run.py` 后即可同时访问前端页面和后端 API。

这种设计兼顾了开发清晰度和部署便利性：

- 前端代码集中在 `frontend/`，不与后端业务逻辑混杂。
- 后端接口集中在 `app/api/routes/`，便于接口扩展。
- 不需要额外启动 Node 服务，降低本地演示和课程答辩环境的部署复杂度。
- 前端静态资源由 FastAPI 直接托管，访问路径稳定。

### 1.2 后端按职责分层

项目目录按职责拆分为：

- `app/api/`：API 路由与依赖注入。
- `app/core/`：配置、日志、异常、容器等基础能力。
- `app/services/`：文档、向量、RAG、大模型、语音、存储等业务服务。
- `app/schemas/`：接口请求和响应结构。
- `app/models/`：内部领域对象。
- `app/utils/`：文本处理、ID 生成等通用工具。

这种分层避免了所有逻辑堆在一个文件里，后续可以单独替换文档解析器、向量库、大模型客户端或前端页面。

### 1.3 使用依赖容器统一组装服务

`app/core/container.py` 中的 `AppContainer` 负责集中初始化：

- 文件存储 `FileStore`
- 元数据存储 `MetadataStore`
- PDF 解析器 `PdfParser`
- 文本清洗器 `TextCleaner`
- 文档切片器 `DocumentChunker`
- BGE-m3 向量模型 `BgeM3Embedder`
- Milvus 或内存向量库
- 意图识别器 `IntentAnalyzer`
- OpenAI 兼容大模型客户端
- 语音转写服务
- 文档服务和 RAG 流水线

这样 API 路由只需要通过依赖注入获取容器，不需要关心各服务的创建细节，降低了接口层复杂度。

### 1.4 应用生命周期管理

FastAPI 使用 `lifespan` 管理应用启动和关闭流程：

- 启动时读取配置。
- 启动时初始化日志。
- 启动时创建业务容器。
- 关闭时释放向量库连接和大模型客户端连接。

相比在模块导入阶段随意创建连接，这种方式更适合 Web 服务，也便于后续接入测试环境和生产环境。

## 2. 配置管理优化

### 2.1 使用 Pydantic Settings 统一读取环境变量

`app/core/config.py` 使用 `pydantic-settings` 定义 `Settings` 类，统一管理：

- 应用名称、运行环境、监听地址、端口和调试模式。
- 上传目录、解析结果目录、SQLite 数据库路径和日志目录。
- 向量库后端、Milvus 地址、集合名称和鉴权信息。
- 嵌入模型名称、运行设备、批量大小。
- RAG 检索参数，如 `TOP_K`、`CHUNK_SIZE`、`CHUNK_OVERLAP`。
- LLM 接口地址、API Key、模型、超时、最大输出和温度。
- 语音转写开关和转写模型。
- 水印清理关键词。

配置集中后，部署时只需要修改 `.env`，不需要直接改代码。

### 2.2 支持 `.env.example`

项目提供 `.env.example`，部署人员可以复制为 `.env` 后按环境修改。这样减少了缺少环境变量导致的启动问题，也便于说明项目需要哪些关键配置。

### 2.3 自动创建运行目录

`get_settings()` 中会自动创建：

- 上传目录
- 解析结果目录
- SQLite 数据库父目录
- 日志目录

这避免了首次运行时因为 `data/uploads`、`data/parsed`、`data/documents` 或 `data/logs` 不存在而报错。

### 2.4 支持 CPU 和 CUDA 切换

嵌入模型设备由 `EMBEDDING_DEVICE` 控制，可以设置为：

- `cpu`：适合普通电脑、本地演示、无显卡服务器。
- `cuda`：适合有 NVIDIA GPU 的服务器，加快向量化速度。

这种配置化设计让同一套代码可以适配不同硬件环境。

## 3. 文档上传与管理优化

### 3.1 上传文件类型校验

`DocumentService.ingest_pdf()` 中明确校验上传文件必须是 `.pdf`，非 PDF 文件会返回业务错误，避免无效文件进入解析流程。

### 3.2 文档 ID 稳定生成

上传后会根据文件名生成 `doc_id`，并基于 `doc_id` 存储原始 PDF 和解析 JSON。相比直接使用原文件名，能够减少中文、空格、重复名称带来的路径问题。

### 3.3 原始文件和解析结果分开保存

项目将文件分为两类保存：

- `data/uploads/`：保存原始 PDF。
- `data/parsed/`：保存解析后的 JSON。

这样可以保留原始材料，同时保留结构化解析结果，便于排查解析质量、重新切片或做离线分析。

### 3.4 文档元数据持久化

项目使用 SQLite 保存文档元信息，包括：

- `doc_id`
- 文件名
- 原始文件路径
- 解析结果路径
- 文档状态
- 页数
- 切片数
- 解析错误
- 自定义元数据
- 创建时间和更新时间

这样即使应用重启，文档列表仍然可以恢复，不依赖内存状态。

### 3.5 支持文档删除

文档删除时会同步处理：

- 删除向量库中的文档片段。
- 删除 SQLite 元数据。
- 删除原始 PDF。
- 删除解析 JSON。

这避免了只删除界面记录但向量库仍残留旧数据的问题。

### 3.6 失败状态记录

文档解析或入库失败时，系统会记录 `failed` 状态和错误信息，便于排查问题，而不是静默失败。

## 4. PDF 解析优化

### 4.1 使用 pdfplumber 提取文本

`PdfParser` 使用 `pdfplumber` 按页读取 PDF 文本，适合处理包含文字层的 PDF。

### 4.2 支持表格抽取

项目不仅提取正文，还会调用 `extract_tables()` 抽取表格，并将表格行转换为 `|` 分隔的文本格式。

对 RAG 系统来说，表格内容如果不处理，很多数字、统计项和对比信息会丢失。该优化使系统可以回答更多表格类问题。

### 4.3 保留页码

解析时为每一页保留 `page_number`。后续切片、检索结果和前端引用都会携带页码，方便用户定位答案来源。

### 4.4 表格解析异常隔离

表格抽取失败时，解析器会记录异常，但不会导致整个 PDF 解析流程直接中断。这样可以提升系统对复杂 PDF 的容错能力。

## 5. 文本清洗优化

### 5.1 水印关键词清理

`TextCleaner` 支持通过 `WATERMARK_PATTERNS` 配置水印关键词，例如机密、仅供内部使用、Confidential 等。解析后会替换这些水印词，减少无意义文本对向量检索的干扰。

### 5.2 页码和控制字符清理

清洗逻辑会处理常见页码文本和不可见控制字符，避免这些字符进入向量模型和大模型上下文。

### 5.3 空白归一化

通过 `normalize_whitespace` 对空格、换行等进行整理，让切片文本更规整，减少因为格式混乱导致的检索效果下降。

### 5.4 行级去重

合并页面文本时会执行行级去重，适合清理页眉、页脚、重复声明等内容。这样可以降低重复片段在检索结果中的占比。

### 5.5 正文与表格合并清洗

每页会将正文和表格合并后统一清洗，再进入切片流程。这样保证表格数据也可以参与向量化和检索。

## 6. 文档切片优化

### 6.1 按页切片

项目以页为单位进行切片，切片记录中保留页码。这样检索返回片段时可以明确指向具体页，引用更可信。

### 6.2 支持滑动窗口重叠

`DocumentChunker` 使用 `CHUNK_SIZE` 和 `CHUNK_OVERLAP` 控制切片长度与重叠长度。

重叠切片可以缓解重要上下文刚好落在切片边界的问题，提高召回质量。

### 6.3 切片携带完整元数据

每个切片包含：

- `chunk_id`
- `doc_id`
- `text`
- `page`
- `chunk_index`
- `source_file`
- `metadata`

这些字段不仅便于检索，也便于前端展示引用来源和后续调试。

### 6.4 空文本跳过

切片前会跳过空白页面或空白文本，减少无意义向量写入，节省向量库空间和检索成本。

## 7. 向量化优化

### 7.1 使用 BGE-m3 嵌入模型

项目使用 `sentence-transformers` 加载 BGE-m3 模型进行文本向量化。BGE-m3 适合多语言和中文语义检索，适合 PDF 中文知识库问答场景。

### 7.2 模型懒加载

`BgeM3Embedder` 在第一次真正调用模型时才加载，而不是应用启动时立即加载。这样可以减少启动阶段失败面，也让配置、健康检查等轻量操作更快。

### 7.3 批量向量化

文档入库时通过 `embed_texts()` 批量处理切片，批量大小由 `EMBEDDING_BATCH_SIZE` 控制。相比逐条向量化，批处理可以更好利用 CPU/GPU 资源。

### 7.4 向量归一化

模型编码时开启 `normalize_embeddings=True`，配合余弦相似度检索，有利于相似度分数稳定。

## 8. 向量库优化

### 8.1 支持 Milvus

项目实现了 `MilvusVectorStore`，用于将切片向量写入 Milvus，适合真实知识库场景下的大规模检索。

### 8.2 自动创建集合和索引

首次写入或检索时，如果 Milvus 中不存在集合，系统会自动创建：

- 主键字段 `chunk_id`
- 文档 ID 字段 `doc_id`
- 页码字段 `page`
- 切片序号 `chunk_index`
- 来源文件名 `source_file`
- 原文片段 `text`
- 向量字段 `embedding`
- COSINE 指标的向量索引

这减少了人工初始化 Milvus 集合的步骤。

### 8.3 支持文档级过滤

检索时可以传入 `doc_ids`，系统会构造 Milvus 过滤表达式，只在选定文档中搜索。前端多选文档问答就是基于这个能力实现的。

### 8.4 支持内存向量库

当 `VECTOR_BACKEND=memory` 时，项目会使用 `InMemoryVectorStore`。这对以下场景很有用：

- 本地快速演示。
- 没有安装 Milvus 的开发环境。
- 调试 RAG 流程。
- 小数据量临时验证。

### 8.5 向量库抽象接口

`BaseVectorStore` 定义统一接口：

- `upsert_chunks`
- `search`
- `delete_by_doc_id`
- `close`

后续如果要接入 FAISS、Qdrant、Elasticsearch、PGVector，可以在不大改 RAG 流程的情况下扩展。

### 8.6 写入前删除重复 chunk

Milvus 写入前会按 `chunk_id` 删除已有记录，再插入新数据，降低重复写入导致的检索污染风险。

## 9. RAG 问答优化

### 9.1 问题归一化

`IntentAnalyzer` 会先对用户问题做基础归一化，去掉部分口语化前缀并整理空白字符，让检索 query 更稳定。

### 9.2 意图识别

系统根据关键词将问题分为：

- `qa`：普通问答
- `definition`：定义解释
- `summarization`：总结摘要
- `table_lookup`：表格数据查询

虽然当前是轻量规则实现，但已经为后续接入更复杂意图识别打好结构基础。

### 9.3 基于归一化问题生成查询向量

RAG 流水线使用归一化后的问题生成查询向量，减少口语噪声对向量检索的影响。

### 9.4 Top-K 检索可配置

默认 `TOP_K` 从配置读取，接口请求也可以传入 `top_k` 覆盖。这样不同问题可以灵活调整召回片段数量。

### 9.5 Prompt 携带结构化证据

构造 Prompt 时，系统会按片段编号组织：

- 来源文档
- 页码
- 相似度分数
- 片段正文

这样大模型能更清楚地看到证据来源，也方便生成可追溯回答。

### 9.6 系统提示约束大模型

系统提示中要求模型严格基于提供的文档片段回答，证据不足时需要说明依据不足。该设计可以降低幻觉回答风险。

### 9.7 返回引用片段

接口不仅返回最终答案，还返回检索到的引用片段。前端可以展示来源页码和相似度，提升用户对答案的信任度。

## 10. 大模型调用优化

### 10.1 使用 OpenAI 兼容接口

`OpenAICompatibleLLMClient` 使用 OpenAI SDK，但 `base_url` 可配置，因此不仅可以调用 OpenAI，也可以调用兼容 OpenAI API 的第三方模型服务或本地网关。

### 10.2 支持超时配置

`LLM_TIMEOUT` 控制请求超时时间，避免模型接口长时间卡住导致 Web 请求无限等待。

### 10.3 支持重试机制

`chat()` 方法通过 `tenacity` 增加指数退避重试，最多重试 3 次。网络抖动或临时接口失败时，系统有机会自动恢复。

### 10.4 输出参数可配置

项目支持配置：

- 模型名称
- 最大输出 token
- temperature
- base_url
- API Key

这使得同一项目可以在低成本模型和高质量模型之间切换。

## 11. 语音输入优化

### 11.1 前端直接录音

前端使用浏览器 `MediaRecorder` 获取麦克风音频，不需要额外安装桌面客户端。

### 11.2 后端统一转写

前端将音频通过 `/api/speech/transcribe` 上传，后端调用大模型语音转写接口返回文本，然后填入问题输入框。

### 11.3 音频不落盘

后端读取音频字节后直接转写，没有保存音频文件，降低了隐私风险和磁盘占用。

### 11.4 自动识别常见音频扩展

前端根据 MIME 类型推断 `webm`、`ogg`、`mp4`、`mp3`、`wav` 等扩展名，提高不同浏览器下的兼容性。

## 12. API 设计优化

### 12.1 路由按业务模块拆分

接口分为：

- `documents.py`
- `chat.py`
- `speech.py`
- `health.py`

这种拆分清晰表达业务边界，后续维护更方便。

### 12.2 使用 Pydantic Schema 约束响应

接口响应由 `app/schemas/` 中的模型定义，能够提升返回结构的一致性，也方便前端按照固定字段渲染。

### 12.3 统一依赖注入

路由函数通过 `Depends(get_container)` 获取容器，不需要在每个接口里重复创建服务对象。

### 12.4 文档内容查询接口

`GET /api/documents/{doc_id}/content` 可以读取解析后的 JSON，方便查看文档解析结果，也便于调试数据质量。

## 13. 前端交互优化

### 13.1 单页面完成主要流程

前端页面支持：

- 上传 PDF
- 查看文档列表
- 删除文档
- 选择文档范围
- 输入问题
- 语音输入
- 查看答案
- 查看引用片段

用户无需切换多个页面即可完成 RAG 知识库问答主流程。

### 13.2 请求封装

前端封装了 `request()` 方法统一处理：

- API 前缀
- JSON 解析
- HTTP 错误
- 后端 `success=false` 业务错误

这减少了每个交互中重复写错误处理逻辑。

### 13.3 上传状态提示

上传时会展示解析中、上传完成、失败等状态，用户能知道当前操作是否仍在进行。

### 13.4 问答按钮禁用

问答请求发起后会禁用提交按钮并显示生成状态，避免用户重复提交导致多次请求。

### 13.5 支持回车提交

问题输入框支持按 Enter 提交、Shift+Enter 换行，符合常见聊天输入体验。

### 13.6 引用片段 HTML 转义

展示引用片段时通过 `escapeHtml()` 处理文本，降低片段内容中包含 HTML 时影响页面结构的风险。

### 13.7 文档筛选问答

前端文档选择框支持选择特定文档，后端据此过滤向量检索范围，使问答更加精准。

## 14. 日志与异常优化

### 14.1 统一业务异常

项目定义了 `AppError`，业务错误可以携带：

- 错误消息
- HTTP 状态码
- 详细信息

接口层会返回统一 JSON 结构，前端更容易处理。

### 14.2 全局未处理异常捕获

未预期异常会被 `unhandled_error_handler` 捕获，返回 500 JSON 响应，并记录异常类型，避免后端直接暴露堆栈给前端。

### 14.3 请求日志中间件

每个 HTTP 请求都会记录开始和结束日志，包括方法、路径和状态码，便于排查接口耗时和失败位置。

### 14.4 滚动文件日志

日志使用 `RotatingFileHandler`，单个日志文件最大 5MB，保留 5 份备份。这样可以避免长期运行时日志无限增长占满磁盘。

### 14.5 降低第三方库日志噪声

项目将 `httpx`、`httpcore`、`openai`、`pymilvus`、`multipart` 等第三方库日志级别调高，重点保留应用自身日志。

## 15. 部署与运维优化

### 15.1 单命令启动

项目入口为 `run.py`，读取配置后启动 `uvicorn`。部署脚本准备完环境后可以直接运行：

```bash
python run.py
```

### 15.2 支持 reload 开关

`DEBUG=true` 时 uvicorn 会启用 reload，适合开发调试；生产或演示稳定环境可设置 `DEBUG=false`。

### 15.3 支持健康检查

项目有独立健康检查路由，部署后可以通过健康检查接口确认服务是否正常启动。

### 15.4 数据目录固定

所有运行数据集中在 `data/` 下，便于备份、迁移和清理。

### 15.5 支持无 Milvus 演示模式

通过 `VECTOR_BACKEND=memory` 可以在没有 Milvus 的情况下启动和演示核心流程，降低部署门槛。

## 16. 可扩展性优化

### 16.1 向量库可替换

由于向量库已经抽象为 `BaseVectorStore`，后续可以扩展：

- FAISS
- Qdrant
- PGVector
- Elasticsearch Vector
- Redis Vector

### 16.2 大模型可替换

由于 LLM 使用 OpenAI 兼容接口，后续可以接入：

- OpenAI
- Azure OpenAI
- 国内兼容 OpenAI 的模型服务
- 本地模型网关
- 私有化大模型平台

### 16.3 文档解析器可替换

当前使用 pdfplumber，如果后续需要增强扫描件 OCR、复杂表格识别或版面分析，可以在 `PdfParser` 层替换为更强解析器，而不影响 API 和 RAG 主流程。

### 16.4 意图识别可升级

当前意图识别是轻量规则实现，后续可以升级为：

- LLM 意图分类
- 关键词 + 正则混合规则
- 小模型分类器
- 根据意图选择不同 Prompt 模板
- 根据意图调整 Top-K 和检索策略

## 17. 项目亮点总结

当前项目已经具备一个完整 RAG 应用的关键闭环：

1. 文档上传。
2. PDF 正文与表格解析。
3. 文本清洗。
4. 分页切片。
5. 本地嵌入模型向量化。
6. Milvus 或内存向量存储。
7. 用户问题归一化。
8. 意图识别。
9. 向量检索。
10. 带证据片段的 Prompt 构造。
11. OpenAI 兼容大模型问答。
12. 答案与引用返回。
13. 前端可视化展示。
14. 语音输入。
15. 日志、异常和元数据持久化。

从工程角度看，项目重点优化了结构清晰度、配置灵活性、文档解析质量、检索可追溯性、部署便利性和后续扩展能力。
