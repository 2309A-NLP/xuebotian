# RAG QA System 已完成优化点

本文档整理当前项目中已经落地的主要优化点，便于答辩、交付说明和后续维护。

## 1. 分层架构与依赖解耦

- 后端采用 FastAPI，按 `api`、`core`、`schemas`、`services`、`models`、`utils` 分层组织。
- `AppContainer` 统一装配文件存储、元数据存储、PDF 解析、文本清洗、切片、向量化、向量库、LLM、语音转写和 RAG Pipeline。
- 业务逻辑集中在 `services`，路由层只负责请求接入和响应组装，降低后续替换模型、向量库和解析器的成本。

## 2. 配置集中化

- 使用 `pydantic-settings` 从 `.env` 加载配置，统一管理应用端口、数据目录、Milvus、Embedding、LLM、语音和流式输出参数。
- 启动时自动创建上传目录、解析目录、SQLite 元数据目录和日志目录，减少部署时的手工准备步骤。
- 向量后端支持 `milvus` 和 `memory` 两种模式，便于本地调试和服务器部署切换。

## 3. PDF 文档处理优化

- 上传阶段限制仅支持 PDF，避免非预期文件进入解析流程。
- 文档解析后按页保留文本、表格和页码信息，便于问答时返回来源页码。
- 文本清洗支持水印词去除、页码噪声清理、控制字符过滤、空白标准化和重复行去重。
- 解析结果持久化为 JSON，原始文件、解析文件和元数据分离保存，方便管理和排查。

## 4. 切片与向量化优化

- 按页切片并保留 `doc_id`、`page`、`chunk_index`、`source_file` 等元信息，检索结果可追溯。
- 支持 `CHUNK_SIZE` 和 `CHUNK_OVERLAP` 配置，兼顾上下文完整性和检索粒度。
- 使用 BGE-m3 作为本地 Embedding 模型，并支持 `cpu` / `cuda` 设备切换及批量向量化。
- 启动时执行 Embedding 和向量库 warmup，减少首次请求的冷启动延迟。

## 5. Milvus 检索优化

- 自动创建 Milvus Collection、Schema 和向量索引，降低首次部署成本。
- 使用 COSINE 相似度和 AUTOINDEX，适合语义检索场景。
- 支持按 `doc_ids` 过滤检索，可实现指定文档范围内问答。
- 文档删除时同步删除向量、元数据、原始文件和解析文件，避免脏数据残留。

## 6. RAG 问答链路优化

- 问答前先进行意图识别和问题标准化，提高检索输入质量。
- Prompt 中显式注入片段来源、页码和相似度，方便模型基于证据回答。
- 对注入 Prompt 的片段长度做 `PROMPT_CHUNK_CHAR_LIMIT` 限制，避免上下文过长导致成本和延迟失控。
- 系统提示词要求严格基于参考片段回答，证据不足时明确说明，降低幻觉风险。

## 7. 流式输出体验优化

- 提供普通问答接口 `/api/chat` 和 SSE 流式接口 `/api/chat/stream`。
- 流式接口先返回 `meta` 事件，包含标准化问题、意图、引用片段和检索耗时。
- Token 输出按字符阈值、时间阈值和句子边界聚合，平衡实时性与前端渲染频率。
- 流式结束返回 `done` 事件，包含完整答案和各阶段耗时。

## 8. 可观测性与异常处理

- 统一配置日志目录，记录请求开始、请求完成和 RAG 各阶段耗时。
- 自定义 `AppError` 和全局异常处理，保证 API 错误响应一致。
- 文档入库失败时记录失败状态和错误原因，便于后台排查。
- RAG Pipeline 输出 Embedding、检索、LLM 和总耗时，便于定位性能瓶颈。

## 9. 前后端分离但部署简洁

- 后端提供 API，前端使用原生 HTML/CSS/JS 调用接口。
- FastAPI 直接挂载 `frontend/assets` 并返回首页，单服务即可完成演示部署。
- 不依赖前端构建工具，服务器部署步骤更简单。

## 10. 语音输入能力

- 提供语音转写服务，前端录音后由后端调用兼容 OpenAI 接口的语音模型。
- 音频以请求内存字节流方式传输，不落盘保存，降低敏感数据残留风险。
