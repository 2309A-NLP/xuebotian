# 项目已做优化点

## 1. 文档接入与解析链路优化

- 已将文档处理拆成 `上传 -> 后台解析 -> 建索引 -> 就绪/失败` 的异步状态流转，前端可感知处理进度，避免上传请求长时间阻塞。
- 已对正文做水印、页码、页眉页脚和跨页重复 boilerplate 清洗，降低噪声对切块和召回的干扰。
- 已增加断行合并与空白归一化，减少 PDF 抽取后句子被硬换行切碎的问题。
- 已对表格做规范化处理，包含表头识别、多行表头合并、跨页表格拼接和结构化序列化，提升表格类问答的可检索性。
- 已支持图片抽取与视觉描述链路，开启 `VISION_ENABLED=true` 后可把图片、曲线、示意图等视觉内容转成可检索文本。
- 已将正文块、表格块、图片块统一纳入 chunk 流程，并做去重处理，减少重复向量写入。

对应代码：
- `app/services/document/manager.py`
- `app/services/document/cleaner.py`
- `app/services/document/table_processor.py`
- `app/services/document/chunker.py`
- `app/services/document/image_processor.py`

## 2. 检索召回质量优化

- 已加入问题归一化和意图识别，把问题区分为普通问答、表格检索、图片检索、定义类、总结类等路线。
- 已加入 LLM 查询改写，并带有安全校验，避免改写时平白新增数字、年份、否定条件或结论。
- 已准备规则兜底改写逻辑，LLM 改写失败时仍能生成适合检索的查询变体。
- 已采用多查询变体并行召回，而不是只用原问题单路检索。
- 已支持 Dense 向量检索与 BM25 关键词检索融合，用 Weighted RRF 做混合排序，兼顾语义匹配和精确词命中。
- 已扩大召回候选池，再进入 rerank，降低首轮召回过窄导致的漏召回问题。
- 已按问题特征对机构名、标题锚点、表格标题、图片 caption、关键词匹配等做软加权。
- 已支持按 `doc_ids` 限定检索范围，适合多文档场景下做定向问答。

对应代码：
- `app/services/intent/analyzer.py`
- `app/services/rag/pipeline_query.py`
- `app/services/rag/pipeline_retrieval.py`
- `app/services/rag/pipeline_ranking.py`
- `app/services/vector/milvus_store.py`

## 3. 生成质量与回答稳定性优化

- 已将证据片段按问题关键词做压缩裁剪，尽量保留命中行、表头、页码、图片标题等高价值上下文，减少无关文本占满 prompt。
- 已针对表格类和图片类内容保留更完整片段，减少结构化证据在裁剪时被破坏。
- 已增加“抽取式回答”兜底策略，当模型第一次回答过于保守时，会重新走一轮更强调证据抽取的 prompt。
- 已增加“原问题重试”机制，如果改写后的查询没有形成可回答证据，会退回原问题再检索一轮。
- 已输出检索引用、问题归一化结果、耗时指标和流式 token，方便前端展示和排障。

对应代码：
- `app/services/rag/pipeline.py`
- `app/services/rag/pipeline_prompting.py`

## 4. 工程化与可运维性优化

- 已通过 `AppContainer` 统一装配解析、向量化、检索、LLM、语音等服务，降低模块耦合。
- 已支持 `milvus` 与 `memory` 两种向量后端，方便开发调试和降级运行。
- 已用 SQLite 保存文档元数据，上传记录、状态、分块数、解析失败原因都可追踪。
- 已加入请求级 `request_id`、文档级 `doc_id` 日志上下文，便于定位线上问题。
- 已对 LLM 请求、流式输出、视觉描述做重试处理，提升外部依赖不稳定时的成功率。
- 已将语音转写作为独立服务接入，不影响主问答链路。

对应代码：
- `app/core/container.py`
- `app/core/logging.py`
- `app/core/log_context.py`
- `app/services/storage/metadata_store.py`
- `app/services/llm/client.py`
- `app/services/speech/transcriber.py`

## 5. 当前仍建议继续补强的点

- `.env.example` 还没有完整覆盖 `rerank`、`query rewrite`、`hybrid search` 等全部配置项，建议后续补齐。
- Linux 部署时不要直接沿用默认 `RERANK_MODEL_NAME`，当前默认值仍偏向本地 Windows 路径。
- 目前缺少正式测试脚本与 CI，后续建议补充 ingest、检索、接口级自动化验证。
- 若生产环境文档量较大，建议进一步补上离线重建索引脚本、模型缓存目录和资源监控。
