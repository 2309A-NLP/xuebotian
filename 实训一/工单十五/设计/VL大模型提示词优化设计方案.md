# RAGFlow 项目 VL 大模型提示词优化设计方案

## 1. 目标与范围

本方案面向 RAGFlow 项目中图片、图表、界面截图、扫描页等视觉内容的 VLM 解析链路，目标是提升以下三类能力：

1. 提高图片转文本结果的可检索性与结构稳定性。
2. 降低视觉模型对图表、流程图、界面截图的幻觉和误判。
3. 让产出的文本更适合进入后续切片、Embedding、召回和问答流程。

本次优化对象聚焦于现有提示词模板与接入方式，不改动模型服务框架本身。

## 3. 现有提示词的主要问题

### 3.1 页面转录提示词过于偏 OCR

`vision_llm_describe_prompt.md` 当前核心目标是“把 PDF 页面图像转成 Markdown”，优点是适合文本页，但对下列内容不够稳定：

1. 图文混排页面中图片区域会被忽略或错误转成正文。
2. 含表格、公式、流程图的页面容易出现结构失真。
3. 对“无法识别时返回空串”的要求过强，可能导致有效弱信号被直接丢弃。

### 3.2 图像类提示词对输出一致性约束不足

`vision_llm_figure_describe_prompt*.md` 已经区分了“结构化数据图”和“普通图片”，这是正确方向，但仍有几个缺口：

1. 对图表数据抽取没有要求“只提取能明确读到的值”，容易补全不存在的数据。
2. 对坐标轴、单位、图例、误差线、趋势线等细节没有优先级规则。
3. 对 UI 截图、流程图、框架图、设备照片、实验图的描述边界还不够清晰。
4. 输出虽然区分两种模式，但没有统一的字段完整性要求，后续索引质量会波动。

### 3.3 缺少“检索友好型”设计

当前提示词更偏“看图说话”，但进入 RAG 后还需要满足：

1. 关键词可召回。
2. 数值与单位可比对。
3. 图名、标题、图例、部件标签可被准确召回。
4. 上下文引用尽量少但关键锚点要保留。

如果提示词只追求自然语言描述，不追求结构化锚点，召回与重排会变差。

## 4. 优化设计原则

### 4.1 先判型，再输出

所有视觉提示词统一采用两段式策略：

1. 先做视觉内容判型。
2. 再进入对应输出模板。

推荐至少区分以下五类：

1. 纯文本页
2. 表格/图表类
3. 普通插图/照片类
4. 界面截图/系统页面类
5. 流程图/结构图/示意图类

原因是这五类内容对“准确性、字段结构、可检索性”的要求明显不同。

### 4.2 只输出可见证据，不补全

提示词必须强化三条硬约束：

1. 只写图中可见内容。
2. 读不清就标记“不可辨认”，不要猜。
3. 不从上下文补全图中不存在的信息。

这是抑制 VLM 幻觉最有效的方式。

### 4.3 输出要服务 RAG，而不是服务可读性

视觉结果进入知识库后，最重要的不是文案好看，而是：

1. 字段稳定
2. 锚点明确
3. 数值保真
4. 标题可召回
5. 图例可召回

所以提示词应优先产出“半结构化文本”，而不是纯自然段落。

### 4.4 弱上下文辅助，强边界约束

对 `vision_llm_figure_describe_prompt_with_context.md` 建议保留上下文，但必须强调：

1. 上下文只用于术语消歧。
2. 不得把上下文信息当成图中事实写入结果。
3. 图中未出现的数值、结论、关系不得补入。

## 5. 目标输出结构设计

### 5.1 文本页

适合保持 Markdown 转录，但增加以下策略：

1. 标题、列表、表格保留原结构。
2. 图片占位区如果存在图题或图号，要保留。
3. 无法识别的小块区域不影响整页输出。

### 5.2 图表/表格类

推荐固定字段：

1. `Visual Type`
2. `Title`
3. `Axes / Legends / Labels`
4. `Data Points`
5. `Captions / Annotations`
6. `Unreadable or Uncertain Parts`

其中 `Data Points` 只允许填可明确识别的数值、类别和单位。

### 5.3 普通图片/照片类

推荐输出：

1. 场景主体
2. 关键对象
3. 可见文字
4. 空间关系
5. 标注或编号

不输出推测性的用途、意图、机制。

### 5.4 UI 截图类

推荐突出：

1. 页面标题
2. 区块布局
3. 菜单、按钮、输入框、表单项
4. 关键状态词与提示词
5. 可见数值、列表项、标签

这样更利于工单、系统说明、运维截图类资料召回。

### 5.5 流程图/结构图类

推荐输出：

1. 节点名称
2. 连接方向
3. 箭头文本
4. 分组/泳道/模块标题
5. 节点间显式关系

不解释流程含义，只记录图上明确呈现的关系。

## 6. 推荐优化策略

### 6.1 提示词体系分层

建议把视觉提示词按用途拆成三层：

1. 页面转录层
   - 用于整页 PDF 图片转 Markdown
2. 图像理解层
   - 用于 figure、chart、screenshot、photo
3. 结构增强层
   - 用于上下文消歧、字段约束、错误抑制

当前项目已有对应雏形，建议继续沿用现有三个模板文件，不新增复杂调用链。

### 6.2 增加“不确定信息显式标记”

相比直接留空，建议在结构化输出里允许：

1. `Unreadable`
2. `Not visible`
3. `Uncertain`

这样既能压制幻觉，也能保留“这张图存在但读不全”的索引信号。

### 6.3 图表输出加入数值优先级

顺序建议固定为：

1. 标题
2. 轴名称与单位
3. 图例
4. 类别标签
5. 明确数值点
6. 注释文字

这比先写长描述更适合后续向量检索。

### 6.4 普通图片输出加入顺序约束

建议明确：

1. 从整体到局部
2. 从左到右、从上到下
3. 先对象后文字
4. 先静态布局后连接关系

顺序稳定后，同类图片的索引文本更可控。

## 7. 落地方式

### 7.1 最小改动落地

不改 Python 逻辑，只替换三份提示词模板：

1. `rag/prompts/vision_llm_describe_prompt.md`
2. `rag/prompts/vision_llm_figure_describe_prompt.md`
3. `rag/prompts/vision_llm_figure_describe_prompt_with_context.md`

优点：

1. 风险低
2. 上线快
3. 易回滚
4. 不影响模型服务接口

### 7.2 配置化落地

通过 `parser.py` 中已有 `system_prompt` 字段，让不同知识库或不同解析任务可选择不同视觉提示词。

建议后续配置维度：

1. 通用文档模式
2. 工业图纸/设备资料模式
3. UI 截图模式
4. 论文图表模式

## 8. 评估指标

建议用 50 到 100 张样本图评估，覆盖论文页、报表图、流程图、系统截图、扫描件。

核心指标：

1. OCR 保真率
2. 图表数值准确率
3. 图例/标题召回率
4. 幻觉率
5. 检索命中率
6. 问答引用正确率

## 9. 风险与控制

### 9.1 风险

1. 提示词更严格后，输出可能变短。
2. 多字段结构会增加部分 token 消耗。
3. 不同 VLM 对“严格格式”的服从度存在差异。

### 9.2 控制措施

1. 先灰度替换提示词模板。
2. 保留原始模板备份。
3. 分知识库验证召回和问答效果。
4. 对图表类优先验证数值幻觉是否下降。

## 10. 结论

本项目的 VL 提示词优化重点不在“写得更像人话”，而在“让视觉结果更稳定、更保真、更适合 RAG”。  
建议优先采取“仅替换模板文件”的低风险方案，再根据业务场景逐步增加模式化配置。
